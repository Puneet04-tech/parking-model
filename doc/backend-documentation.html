System Overview

The Parking Intelligence backend consists of a machine learning pipeline that processes parking violation data, trains predictive models, and serves them through a FastAPI REST API. The system uses LightGBM for model training, advanced feature engineering for data preparation, and semi-supervised learning to leverage unlabeled data.

Backend Architecture

The backend follows a modular architecture with clear separation between data processing, model training, and inference serving:

Data Layer

CSV data ingestion, cleaning, and feature engineering pipeline

ML Layer

LightGBM models with hyperparameter tuning and semi-supervised learning

API Layer

Cell 3: Data Loading

Purpose: Load the parking violation dataset from CSV file.

Functionality: Reads 298,450 rows with 24 columns including location, vehicle details, violation types, timestamps, and validation status. Displays dataset info and structure.

Key Columns: latitude, longitude, vehicle_type, violation_type, created_datetime, police_station, junction_name, validation_status

Cell 4: Data Cleaning

Purpose: Handle missing values and filter spatial outliers.

Operations:

  • Filters coordinates to Bangalore bounding box (lat: 12.8-13.3, lon: 77.4-77.8)
  • Fills missing location with 'UNKNOWN'
  • Fills missing center_code with -1
  • Fills missing police_station and junction_name

Rationale: Spatial filtering removes GPS errors and data from outside the city. Missing value handling ensures no nulls break the ML pipeline.

Cell 5: Violation Type Parsing

Purpose: Parse JSON-formatted violation types and create binary indicators.

Functionality:

  • Parses violation_type and offence_code JSON strings into lists
  • Identifies all 27 unique violation types in dataset
  • Creates binary indicators for top 15 most frequent violations
  • Counts total violations per report

Output: 15 new binary columns (e.g., violation_wrong_parking, violation_no_parking) and violation_count column.

Cell 6: Temporal Feature Engineering

Purpose: Extract time-based features from timestamps.

Features Created:

  • hour, dayofweek, month, minute - Extracted from datetime
  • is_weekend - Binary flag for weekends
  • hour_sin, hour_cos - Cyclic encoding for continuous time representation
  • pincode - Extracted from location text
  • loc_has_metro, loc_has_cross, loc_has_road, loc_has_near, loc_has_opp - Text-based location features

Rationale: Cyclic encoding captures that hour 23 and hour 0 are close in time. Text features capture location context without requiring geocoding.

Cell 7: Frequency Encoding

Purpose: Convert high-cardinality categorical features to numeric using frequency counts.

Features: vehicle_type_freq, police_station_freq, junction_name_freq, pincode_freq

Method: Count occurrences of each category and map to frequency. Helps tree-based models learn density patterns.

Cell 8: Target Encoding (OOF)

Purpose: Calculate user and device reliability scores using Out-of-Fold target encoding.

Method:

  • Uses 5-fold cross-validation to prevent data leakage
  • Calculates approval rate for each user/device in training folds
  • Applies Laplace smoothing (smoothing factor = 10) to handle rare users/devices
  • Maps encodings back to full dataset with global mean as fallback

Output: user_target_enc, device_target_enc - Values between 0-1 indicating historical reliability.

Importance: Captures reporter credibility without data leakage, crucial for identifying reliable vs. spam reporters.

Cell 9: Model 1 Definition

Purpose: Define feature set and train initial Model 1 (Report Validator).

Features: 42 features excluding leakage columns (updated_vehicle_*, validation_timestamp, etc.)

Model: LightGBM classifier with regularization (reg_alpha=2.0, reg_lambda=10.0, max_depth=6)

Training: 80-20 train-test split with stratification, early stopping after 30 rounds.

Performance: 75.77% accuracy, 75.82% ROC-AUC on test set.

Cell 10: Enhanced Feature Engineering

Purpose: Add 13 advanced features to improve Model 1 performance.

New Features:

  • dist_from_center - Euclidean distance from Bangalore city center
  • user_report_count - Cumulative count of reports per user
  • last_report_gap_hours - Time since user's last report
  • device_report_count - Cumulative count per device
  • pincode_report_density - Reports in same pincode
  • has_multiple_violations - Binary flag for >1 violations
  • hour_x_weekend - Interaction between hour and weekend
  • lat_x_lon - Interaction between coordinates
  • hour_sin_x_cos - Interaction of cyclic time features
  • police_station_activity - Cumulative reports per station
  • junction_complexity - Word count in junction name
  • is_morning_peak - Binary for 8-10am
  • is_evening_peak - Binary for 5-7pm

Rationale: These features capture user behavior patterns, location complexity, and temporal interactions that improve model discrimination.

Cell 11: Model 1 Retraining with Enhanced Features

Purpose: Retrain Model 1 with 55 features and class imbalance handling.

Changes:

  • Updated to 55 features (42 original + 13 enhanced)
  • Added scale_pos_weight for class imbalance (negative_samples / positive_samples)
  • Increased model capacity (max_depth=8, num_leaves=63)

Performance: 73.58% accuracy, 79.98% ROC-AUC - improved ROC-AUC by 4.16% despite lower accuracy due to better class balance handling.

Cell 12: Hyperparameter Tuning with Optuna

Purpose: Automated hyperparameter optimization to find best Model 1 configuration.

Method:

  • Optimizes 9 hyperparameters: n_estimators, learning_rate, max_depth, num_leaves, min_child_samples, subsample, colsample_bytree, reg_alpha, reg_lambda
  • Uses 3-fold cross-validation for faster evaluation
  • Runs 40 trials to find optimal configuration

Cell 13: Semi-Supervised Learning (SSL)

Purpose: Leverage 125k+ unlabeled reports using pseudo-labeling to improve model performance.

Method:

  • Uses enhanced Model 1 to predict labels for unlabeled data

  • Retrains model on combined dataset

Performance: Achieves 80.76% accuracy, 84.07% ROC-AUC - 5% improvement over supervised model.

Importance: Dramatically increases training data without manual labeling, leveraging the large volume of unlabeled reports.

Result: LightGBM outperforms NN for this tabular data task. NN skipped if TensorFlow not installed.

Cell 15: Ensemble Model

Method: Weighted averaging of predictions from multiple models.

Model 1: Report Validator (Detailed)

Purpose

Binary classifier that determines if a parking violation report is legitimate (APPROVED) or should be rejected (REJECTED). This filters out spam and false reports before officers investigate.

Algorithm

LightGBM (Light Gradient Boosting Machine) - a gradient boosting framework that uses tree-based learning algorithms.

Why LightGBM?

  • Fast training speed compared to other boosting algorithms
  • Handles categorical features natively
  • Good performance on tabular data
  • Memory efficient
  • Built-in regularization to prevent overfitting

Feature Set (55 Features)

Spatial (4)

latitude, longitude, dist_from_center, pincode

Temporal (7)

hour, dayofweek, month, minute, is_weekend, hour_sin, hour_cos

Vehicle (3)

vehicle_type, vehicle_type_freq, description

Location (6)

police_station, junction_name, center_code, loc_has_metro, loc_has_cross, loc_has_road, loc_has_near, loc_has_opp

Violations (15)

Binary indicators for 15 violation types + violation_count

Frequency (4)

police_station_freq, junction_name_freq, pincode_freq, vehicle_type_freq

Target Encoding (2)

user_target_enc, device_target_enc

Behavioral (4)

user_report_count, last_report_gap_hours, device_report_count, pincode_report_density

Interaction (3)

hour_x_weekend, lat_x_lon, hour_sin_x_cos

Complexity (2)

police_station_activity, junction_complexity

Peak Hours (2)

is_morning_peak, is_evening_peak

Training Process

  1. Data Preparation: Filter labeled reports (approved/rejected), convert categorical columns to category dtype
  2. Feature Engineering: Apply all 55 features with proper data types
  3. Train-Test Split: 80-20 split with stratification to maintain class balance
  4. Class Imbalance: Calculate scale_pos_weight = negative_samples / positive_samples
  5. Model Training: LightGBM with 500 estimators, early stopping after 50 rounds
  6. Semi-Supervised: Add high-confidence pseudo-labels from unlabeled data
  7. Final Training: Retrain on combined labeled + pseudo-labeled dataset

Model 2: Spatial Hotspot Detector (Detailed)

Purpose

DBSCAN (Density-Based Spatial Clustering of Applications with Noise) clustering algorithm that identifies chronic violation zones and spatial hotspots. This helps officers understand where violations are concentrated and plan patrol routes more effectively.

Algorithm

DBSCAN is a density-based clustering algorithm that groups points that are closely packed together while marking points in low-density regions as outliers.

Why DBSCAN?

  • No Predefined Clusters: Unlike K-means, DBSCAN doesn't require specifying the number of clusters in advance
  • Handles Arbitrary Shapes: Can discover clusters of any shape, not just spherical
  • Noise Detection: Automatically identifies and filters outliers as noise
  • Density-Based: Groups violations based on actual density, not just distance
  • Robust to Outliers: Outliers don't affect cluster formation

How DBSCAN Works

  1. Core Points: A point is a core point if it has at least min_samples points within eps distance
  2. Border Points: Points within eps distance of a core point but with fewer than min_samples neighbors
  3. Noise Points: Points that are neither core nor border points
  4. Cluster Formation: All density-reachable points from a core point form a cluster

Parameters

eps: 0.005 (epsilon - neighborhood radius in degrees)
min_samples: 10 (minimum points required to form a cluster)

eps Selection: 0.005 degrees ≈ 550 meters in Bangalore latitude. This captures violations that are within walking distance of each other.

min_samples Selection: 10 violations minimum ensures we only identify significant hotspots, not random coincidences.

Input Data

Uses latitude and longitude coordinates from all parking violation reports (298,450 points after cleaning).

Output

  • Cluster Labels: Each violation assigned a cluster ID (-1 for noise/outliers)
  • Hotspot Coordinates: Centroid coordinates for each identified cluster
  • Cluster Sizes: Number of violations in each hotspot
  • Visualization: Spatial maps showing hotspot distribution

Use Cases

Patrol Route Planning

Officers can prioritize routes through identified hotspots

Resource Allocation

Deploy more officers to areas with dense violation clusters

Infrastructure Planning

Identify areas needing more parking infrastructure

Trend Analysis

Track how hotspots change over time

Implementation in Notebook

DBSCAN Cell

Code:

from sklearn.cluster import DBSCAN

# Prepare coordinates
coords = df[['latitude', 'longitude']].values

# Apply DBSCAN
dbscan = DBSCAN(eps=0.005, min_samples=10)
clusters = dbscan.fit_predict(coords)

# Add cluster labels to dataframe
df['cluster'] = clusters

# Analyze results
n_clusters = len(set(clusters)) - (1 if -1 in clusters else 0)
n_noise = list(clusters).count(-1)

print(f"Number of clusters: {n_clusters}")
print(f"Number of noise points: {n_noise}")

Output Interpretation:

  • Cluster ID -1 indicates noise/outliers (isolated violations)
  • Cluster IDs 0, 1, 2, etc. represent distinct hotspots
  • Hotspots with more points indicate chronic problem areas

The notebook includes scatter plots showing:

  • Hotspots highlighted with different colors
  • Noise points shown in gray

Model 3: Congestion Priority Predictor (Detailed)

Purpose

Multi-class classifier that predicts enforcement priority (LOW/MEDIUM/HIGH) based on ambient location and time features. Used for quick assessment when full violation details aren't available.

Algorithm

LightGBM multi-class classifier with 3 output classes.

Feature Set (11 Features)

Spatial (2)

latitude, longitude

Temporal (4)

hour_sin, hour_cos, dayofweek, is_weekend, month

Area Density (3)

police_station_freq, pincode_freq, junction_name_freq

Reliability (2)

user_target_enc, device_target_enc

Priority Mapping

0 → LOW: Minimal congestion impact
1 → MEDIUM: Moderate impact, monitor
2 → HIGH: Severe impact, immediate dispatch

Use Case

This model is designed for situations where officers have basic location and time information but lack detailed violation specifics. It provides a quick triage assessment to determine if immediate dispatch is warranted.

Training Data

Trained on historical enforcement data where priority was determined based on actual congestion impact and officer response patterns.

Feature Engineering Details

Cyclic Time Encoding

Time is circular (23:00 is close to 00:00), but linear encoding treats them as far apart. Cyclic encoding using sine and cosine preserves this circular nature:

hour_sin = sin(2 * π * hour / 24)
hour_cos = cos(2 * π * hour / 24)

This allows the model to understand that hour 23 and hour 0 are temporally close.

Target Encoding (Out-of-Fold)

Standard target encoding causes data leakage. OOF prevents this:

  1. Split data into K folds
  2. For each fold, calculate target encoding using only other K-1 folds
  3. Apply encoding to validation fold
  4. Repeat for all folds

Laplace smoothing is applied: (count + smoothing * global_mean) / (total + smoothing)

Frequency Encoding

High-cardinality categorical features (like police_station with 100+ values) are converted to their frequency counts. This captures density patterns without creating hundreds of dummy variables.

Interaction Features

Feature interactions capture non-linear relationships:

  • hour_x_weekend: Captures that weekend evenings have different patterns than weekday evenings
  • lat_x_lon: Captures spatial relationships
  • hour_sin_x_cos: Captures complex temporal patterns

Peak Hour Indicators

Binary flags for known congestion periods:

  • Morning peak: 8-10 AM (commute hours)
  • Evening peak: 5-7 PM (commute hours)
These help the model learn that violations during these times have higher congestion impact.
